# 助盲跑 PRD

## 背景与目标

盲人有跑步锻炼的需求，但缺乏安全跑步的陪伴支持。目前盲人跑步主要依赖跑团在固定地点的定期活动。

**目标**：建立一个连接盲人跑者与志愿者/跑团的平台，让盲人跑步需求的响应更高效、更规模化。

## 产品阶段规划

### 一期：志愿者主导，建立基础

**策略**：先吸引志愿者到平台发布跑步活动，盲人用户再从中选择适配的需求报名。

**核心流程**：
1. 志愿者注册 → 发布跑步活动
2. 盲人浏览活动列表 → 报名参加
3. 活动完成后双方评价

### 二期：跑团接入，规模化活动供给

**核心变化**：跑团作为活动组织者接入，志愿者归属跑团

**志愿者流程**：
1. 注册时选择要加入的跑团
2. 跑团审核志愿者加入申请
3. 审核通过后，志愿者可发布助跑活动

**跑团功能**：
- 跑团管理员发布和管理例跑活动
- 查看每期活动的报名列表（确认/待确认）
- 管理跑团成员（志愿者）

**价值**：跑团提供稳定的活动供给，志愿者归属跑团后有组织背书

### 长期：匹配深化，长期关系建立

**匹配类型**：
- **流动匹配**：单次活动匹配，灵活自由
- **深度搭子**：长期一对一/一对多稳定陪伴关系

**长期建联机制**（规划中）：
- 活动后互加好友
- 长期搭子关系绑定
- 共同参与活动记录

## 用户场景

### 场景 1: 志愿者入驻跑团

**用户类型**：志愿者
**需求**：我想加入一个跑团，参与助盲跑
**流程**：注册 → 选择跑团 → 提交加入申请 → 跑团审核 → 通过后发布活动
**预期结果**：成为跑团成员，开始发布助跑活动

### 场景 2: 跑团发布例跑活动

**用户类型**：跑团管理员
**需求**：我们跑团每周六早上在某公园例跑，欢迎盲人参加
**预期结果**：在平台发布跑团例跑活动，可看到报名列表和确认状态

### 场景 3: 盲人选择活动

**用户类型**：盲人
**需求**：我想跑步，希望有人陪伴
**预期结果**：浏览平台上的跑步活动，选择时间/地点合适的报名参加

### 场景 4: 建立长期搭子关系

**用户类型**：志愿者 / 盲人
**需求**：我们活动匹配过几次，希望建立稳定陪伴关系
**预期结果**：绑定成为长期搭子，后续活动自动匹配

## 功能需求

### P0（必须有）

| 功能 | 描述 | 验收标准 |
|------|------|----------|
| 志愿者注册/登录 | 基础账号体系 | 能创建账号并登录 |
| 跑团注册/管理 | 跑团创建自己的主页 | 跑团能维护基本信息 |
| 加入跑团流程 | 志愿者申请 → 跑团审核 | 审核状态可追踪 |
| 发布跑步活动 | 志愿者/跑团发布活动时间、地点、时长、陪伴方式 | 活动能正常发布并展示 |
| 活动列表 | 盲人浏览所有可参加的活动 | 能按时间/地点筛选 |
| 活动报名 | 盲人报名参加某个活动 | 组织者收到报名通知 |
| 报名状态管理 | 组织者查看报名列表 | 可标记确认/待确认/缺席 |

### P1（最好有）

| 功能 | 描述 | 验收标准 |
|------|------|----------|
| 活动提醒 | 活动开始前提醒双方 | 双方都能收到通知 |
| 评价体系 | 完成后双方互评 | 有评分记录 |
| 盲人发布需求 | 盲人主动发起跑步需求 | 志愿者/跑团可承接 |
| 激励机制 | 志愿者积分、称号、排行榜 | 激励志愿者持续参与 |

### P2（后续考虑）

| 功能 | 描述 |
|------|------|
| 深度搭子绑定 | 长期稳定的一对一/一对多陪跑关系 |
| 搭子匹配推荐 | 基于活动记录推荐长期搭子 |
| 跑团认证体系 | 认证跑团标识，增加可信度 |
| 紧急联系人 | 活动中的紧急联络人功能 |

## 非目标（明确不做的）

- 一期不实现实时陪跑定位追踪
- 一期不实现复杂的钱包/支付系统
- 不做陌生人社交功能
- 不做活动保险自动购买（可引导第三方）

## 激励与安全

### 激励机制

- **积分系统**：完成活动获得积分，可兑换称号
- **称号体系**：新手志愿者 → 活跃志愿者 → 资深志愿者 → 搭子达人
- **排行榜**：按帮助次数/时长，让志愿者有成就感
- **长期搭子奖励**：稳定陪伴关系获得额外奖励

### 安全保证

- **平台背书**：志愿者归属于跑团，跑团有组织责任
- **活动记录**：所有活动在平台留档，可追溯
- **评价监督**：双向评价建立信任
- **身份认证**：跑团认证、志愿者认证（可选）

## 技术约束

- 需要无障碍支持（屏幕阅读器兼容，盲人可用）
- 移动端优先
- 消息通知触达率要保证
- 活动数据需长期保存（信任基础）
